Data Collection Requirements

What data MedBridge must capture, where it comes from, and which items are must-have vs nice-to-have. For team review before Sprint 1 closes.

A — User Identity & Authentication
Data Point Priority Source Purpose
User signup (email, timestamp) MUST HAVE Supabase Auth Signup conversion rate; cohort analysis
Login events (timestamp, method) MUST HAVE Supabase Auth Auth success/failure rate; DAU/WAU calculation
Email verification status MUST HAVE Supabase Auth Funnel drop-off: signup → verified
Preferred language NICE TO HAVE Frontend Localization planning; plain-language summary tuning
Date of birth NICE TO HAVE Frontend Age-adjusted reference ranges for lab results
Onboarding completion timestamp MUST HAVE FastAPI Onboarding funnel completion rate
B — Usage & Session Data
Data Point Priority Source Purpose
Session start / end timestamps MUST HAVE FastAPI Session duration; DAU/WAU; retention
Session duration (seconds) MUST HAVE FastAPI Average session length KPI
Device type (mobile / desktop) NICE TO HAVE Frontend Responsive design prioritization
Entry point (upload / summary / Q&A) NICE TO HAVE Frontend Navigation flow analysis; feature adoption
Days active per user per week MUST HAVE FastAPI WAU and weekly retention calculation
C — Document Upload & Ingestion
Data Point Priority Source Purpose
Upload timestamp MUST HAVE FastAPI Uploads per user per week; growth trend
Source type (PDF upload vs FHIR sync) MUST HAVE FastAPI Channel split; FHIR adoption rate
Ingestion status (success / failure / processing) MUST HAVE FastAPI Pipeline reliability; error rate tracking
File type (PDF, FHIR JSON, etc.) MUST HAVE FastAPI Parser routing and error attribution
File size (bytes) NICE TO HAVE FastAPI Storage growth planning; timeout risk prediction
D — AI / Summary & Q&A Interactions
Data Point Priority Source Purpose
Summary generation timestamp & latency (ms) MUST HAVE Groq API AI latency KPI; SLA monitoring
Model used (e.g. Llama-3.3-70B) MUST HAVE Groq API Cost tracking; model comparison over time
Q&A question count per session MUST HAVE FastAPI AI engagement depth KPI
Q&A answer feedback rating (1–5) MUST HAVE Frontend AI quality; user satisfaction signal
Q&A response latency (ms) NICE TO HAVE Groq API UX quality; Groq performance benchmarking
Summary regeneration count NICE TO HAVE FastAPI Quality signal — repeated summaries may indicate poor output
E — FHIR / EHR Connection Events
Data Point Priority Source Purpose
FHIR OAuth connection created (timestamp) MUST HAVE FHIR OAuth FHIR adoption rate; demo KPI
Last sync timestamp & status MUST HAVE FHIR OAuth Sync reliability; stale data detection
Scopes granted by patient MUST HAVE FHIR OAuth Data completeness; what clinical data is available
Token expiry & refresh events NICE TO HAVE FHIR OAuth Session continuity; silent re-auth success rate

📋 Assumptions — Share with team for feedback

  • No PII beyond email is stored in analytics tables; date of birth and display name stay in user_profiles behind RLS.
  • Session tracking is server-side (FastAPI creates/closes sessions on auth events), not browser analytics (no Mixpanel/Segment).
  • "Transaction data" from the program template is not applicable to MedBridge in the financial sense; document uploads and AI interactions are the equivalent engagement events.
  • All clinical data (lab values, medications, conditions) is sourced from either FHIR or PDF extraction — no manual patient input in Sprint 1.
  • Groq token counts are not tracked in Sprint 1; latency (ms) is the primary AI cost proxy until we have billing data.
  • FHIR scope is Epic public sandbox only for Sprint 1 — no real patient data, synthetic only.

KPI Definitions

12 KPIs across 5 categories. Each is defined with its formula, data source, reporting cadence, and current feasibility assessment for Backend and Frontend to validate.

User Engagement
Daily Active Users (DAU)
Engagement
Count of distinct users who completed at least one authenticated session on a given calendar day.
COUNT(DISTINCT user_id) FROM sessions WHERE DATE(session_start) = target_date
📅 Daily 🗄️ sessions ✓ Feasible
Weekly Active Users (WAU)
Engagement
Count of distinct users with at least one session in a rolling 7-day window.
COUNT(DISTINCT user_id) FROM sessions WHERE session_start >= NOW() - INTERVAL '7 days'
📅 Weekly 🗄️ sessions ✓ Feasible
Weekly Retention Rate
Engagement
% of users active in week N who also had a session in week N+1. Measures stickiness of the core workflow.
WAU(week N+1 returning) / WAU(week N) × 100
📅 Weekly 🗄️ sessions ✓ Feasible
Avg Session Duration
Engagement
Mean session length in minutes across all completed sessions in the reporting period. Proxy for depth of engagement.
AVG(duration_seconds) / 60 FROM sessions WHERE session_end IS NOT NULL
📅 Weekly 🗄️ sessions ✓ Feasible
Authentication
Signup Conversion Rate
Auth
% of users who signed up and also verified their email within 24 hours. Measures top-of-funnel health.
COUNT(email_confirmed_at IS NOT NULL) / COUNT(created_at) × 100 FROM users
📅 Weekly 🗄️ users ✓ Feasible
Auth Success Rate
Auth
% of login attempts that result in a valid session. Surfaces token, OAuth, or PKCE issues early.
SELECT * FROM public.kpi_auth_activity — view live in Supabase, queries auth.audit_log_entries payload JSON
📅 Daily 🗄️ kpi_auth_activity view ✓ View live — populates when auth wired up
AI & Document Intelligence
Summary Generation Rate
AI
% of uploaded health records that result in a successfully generated plain-language summary.
COUNT(summaries.id) / COUNT(health_records.id) × 100 (joined on health_record_id)
📅 Weekly 🗄️ health_records, summaries ✓ Feasible
AI Response Latency (P50 / P95)
AI
Median and 95th-percentile time in ms from upload to summary delivery. Tracks Groq performance and pipeline health.
PERCENTILE_CONT(0.5) / (0.95) WITHIN GROUP (ORDER BY generation_ms) FROM summaries
📅 Daily 🗄️ summaries ✓ Feasible
Q&A Engagement Depth
AI
Average number of user messages per health_record_id in chat_messages. Higher = deeper AI engagement with a document.
AVG(msg_count) FROM (SELECT health_record_id, COUNT(*) AS msg_count FROM chat_messages WHERE role='user' GROUP BY health_record_id)
📅 Weekly 🗄️ chat_messages ✓ Feasible
AI Answer Satisfaction Score
AI
Average user feedback rating (1–5 stars) on Q&A answers. Primary quality signal for Llama 3.3 output on medical context.
AVG(feedback_rating) FROM chat_messages WHERE feedback_rating IS NOT NULL
📅 Weekly 🗄️ chat_messages ⚡ FE ticket open — James to add feedback_rating col + rating widget
Clinical Data Richness
Avg Clinical Tables Populated per Record
Clinical
Average number of clinical tables (lab_results, medications, conditions, encounters, follow_ups, allergies, summaries) with at least 1 row per health_record_id. Measures parser completeness.
SELECT * FROM public.kpi_clinical_completeness -- view queries all 7 clinical tables via UNION COUNT
📅 Per upload 🗄️ kpi_clinical_completeness view ✓ View ready -- run SQL below to create
Abnormal Lab Flag Rate
Clinical
% of lab results flagged as high or low (not normal). Useful for validating parser accuracy against known Synthea data distributions.
COUNT(flag IN ('high','low','critical')) / COUNT(*) × 100 FROM lab_results
📅 Per ingestion run 🗄️ lab_results ✓ Feasible
FHIR / EHR Integration
FHIR Connection Adoption Rate
FHIR
% of registered users who have completed at least one SMART on FHIR OAuth connection. Key demo metric for stakeholders.
COUNT(DISTINCT user_id FROM fhir_connections) / COUNT(DISTINCT id FROM users) × 100
📅 Weekly 🗄️ users, fhir_connections ✓ Feasible
FHIR Sync Success Rate
FHIR
% of FHIR data sync attempts (token refresh + bundle fetch) that completed without error. Validates Epic sandbox reliability.
successful_syncs / total_sync_attempts × 100 (from ingestion_status on health_records where source_type = 'fhir')
📅 Per sync 🗄️ health_records, fhir_connections ✓ Feasible
